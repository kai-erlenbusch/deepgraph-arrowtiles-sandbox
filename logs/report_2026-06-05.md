# Deepgraph WebGPU: Project Report
**Date:** June 5, 2026

## 1. High-Level Purpose & Flow

**Purpose:** 
The primary objective of this project is to build an ultra-high-performance, web-based 2D scatterplot visualization capable of rendering millions of data points (e.g., 2.8 million points from the LMSys dataset) smoothly in the browser. It aims to replicate the aesthetics and scalability of "Deepscatter" by streaming spatial quadtree-chunked data and rendering it using cutting-edge WebGPU capabilities in Three.js.

**High-Level Flow:**
1. **Data Pre-processing:** Raw datasets are ingested by DuckDB (via Python), spatially partitioned into a quadtree structure, and exported as highly compressed Apache Arrow (Feather) files.
2. **Dynamic Streaming:** As the user navigates the canvas, the `TileManager` calculates the viewport bounds and dynamically streams only the Arrow tiles necessary for the current zoom level and position.
3. **WebGPU Rendering:** The raw Arrow buffers are directly mapped into WebGPU `InstancedBufferGeometry`. The Three.js Shading Language (TSL) uses Additive Blending and custom distance functions to draw perfectly anti-aliased glowing circles that maintain a constant pixel size regardless of zoom.

## 2. Chosen Architecture

### Backend & Data Pipeline
- **DuckDB & Python:** Used for heavy-lifting data aggregation and spatial partitioning (`build_quadtree.py`). DuckDB calculates bounds, generates zoom levels, and chunks the Parquet data.
- **Apache Arrow IPC (Feather):** Chosen as the transport format due to its zero-copy deserialization capabilities in the browser, allowing millions of points to be passed to WebGPU instantly.

### Frontend Framework
- **Vite + TypeScript:** Provides a fast, modern build system with strict typing for robustness.
- **Three.js (WebGPU Backend):** The core rendering engine. We bypass the legacy WebGL backend in favor of the new `WebGPURenderer`.

### Rendering Paradigm (Instanced Mesh + TSL)
- **Topological Limitation Bypass:** WebGPU's `point-list` topology does not natively support `gl_PointCoord`. To overcome this, we architected an `InstancedMesh` approach using a single 1x1 `PlaneGeometry` (Quad).
- **TSL Nodes:** We utilize `MeshBasicNodeMaterial` with TSL (Three.js Shading Language). The shader calculates `smoothstep` based on UV coordinates to carve perfect circles out of the quad. 
- **Dynamic Constant Scaling:** We inject the `OrthographicCamera`'s zoom directly as a dynamic uniform (`zoomUniform`) into the vertex shader. By dividing the base size by the zoom uniform, the GPU actively counteracts the camera's projection scaling, ensuring the points remain a constant 2.5 pixels wide no matter how deeply the user zooms.

## 3. Current Progress

We have successfully completed the core foundational phases of the project:

- **[x] Phase 1: Basic Streaming Architecture**
  - Configured Vite and the WebGPU render loop.
  - Implemented the `TileManager` to intelligently traverse the quadtree, parse Arrow IPC streams, and detect viewport intersections.
  - Resolved massive memory spikes by enforcing a strict "level-by-level" fetching logic—preventing deep recursive fetches for empty (404) regions.

- **[x] Phase 2: Deepscatter Aesthetics (Custom Shaders)**
  - Replaced legacy `THREE.Points` with high-performance `InstancedBufferGeometry`.
  - Implemented the custom 5-color categorical palette (Yellows, Oranges, Teals, Blues).
  - Resolved a `BigInt` parsing bug that previously collapsed all points into a single color bucket.
  - Achieved the signature "glowing" heatmap aesthetic via Additive Blending and anti-aliased TSL circular cutouts.
  - Fixed quadratic screen-space explosions by patching the vertex scale divisor.

## 4. Roadmap & Suggested Next Steps

### Phase 3: Interactive Tooling (Next)
- **Raycasting / Hover Metadata:** Implement GPU-accelerated picking (such as a color-picking shader) or screen-space spatial hashing to allow users to hover over individual points and instantly retrieve the underlying metadata (e.g., prompt text, model details) directly from the cached Arrow tables.

### Phase 4: UI & Advanced Controls
- **Filtering & Coloring Options:** Add UI controls to dynamically swap color palettes or filter points based on metadata columns without re-fetching tiles.
- **Minimap / Navigation:** Create a small minimap overlay indicating where the user is currently zoomed within the massive 2.8 million point cloud.

### Edge Cases & Pitfalls to Monitor
- **VRAM Saturation:** While Arrow buffers are efficient, holding too many high-resolution tiles in active memory could crash mobile browsers. We will eventually need to implement an LRU cache eviction policy in `TileManager` to dispose of geometries that are far out of bounds.
- **Browser Compatibility:** WebGPU is still relatively new; maintaining a graceful fallback to WebGL for older browsers could be considered down the line.
