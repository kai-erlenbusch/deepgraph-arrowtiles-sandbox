Your proposed change: *"Update `loadTile` to map over `t.data.buffer` directly instead of `new Uint8Array(t.data)`."*

**The Hidden Danger:** The `pmtiles` library fetches data in "blocks" (often 512KB or larger) to minimize HTTP requests. Multiple small tiles might reside inside a single block. Therefore, the `t.data` object returned by `getZxy()` is often a `Uint8Array` that acts as a *view* into a much larger underlying `ArrayBuffer`.

If you execute a postMessage transfer on `t.data.buffer`, you are forcefully detaching the *entire* block from the main thread. **The "Zero-Allocation" Alternative:**
We don't actually need to eliminate the *copying* of the bytes; we just need to eliminate the *allocation* of new memory to stop the Garbage Collector from thrashing. 

Instead of bypassing the copy, **we should use a Buffer Pool for the raw payloads**.

* **Create a `rawPayloadPool`** in `PMTilesClient.ts` (just like you did for the Arrow columns).
* When a tile arrives, pop a buffer from the pool (or allocate one if empty).
* Copy the bytes natively using `.set()`: `pooledArray.set(t.data)`.
* Pass `pooledArray.buffer` as the Transferable to the worker.
* When the worker is done parsing the Arrow IPC data, it transfers that raw payload buffer *back* to the main thread, where you push it back into `rawPayloadPool`.

This gives you the best of both worlds: PMTiles keeps its cache intact, and you achieve **Zero-Allocation** on the Main Thread, completely eliminating the GC stutters.
